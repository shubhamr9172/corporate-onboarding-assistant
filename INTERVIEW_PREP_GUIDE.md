# 🎓 Corporate Onboarding Assistant V2 — Interview Preparation & System Engineering QA

This document serves as a comprehensive reference guide for your project defense or software engineering interviews. It details the technical challenges, core design decisions, system architectures, and production-hardening solutions built into the **Corporate Onboarding Assistant V2**.

---

## 🏗️ Part 1: High-Level Architecture & LangGraph Orchestration

### Q1: What is the overall architecture of this project, and why did you choose it?
**Answer:**
The project is built as a stateful, multi-turn Retrieval-Augmented Generation (RAG) assistant using a microservices-aligned architecture in Python:
* **Frontend**: Streamlit, custom-styled with a premium glassmorphic Aurora theme.
* **Orchestration**: **LangGraph** (State Machine), managing a compiled `StateGraph` with a SqliteSaver checkpointer.
* **Database & Retrieval**: **ChromaDB** for vector embeddings and document search; **FlashRank** for CPU-based reranking.
* **Caching & Rate Limiting**: **Redis** for L1 exact match caching and fixed-window session rate limiting.
* **Safety**: Custom diacritic-normalized Regex and string-check Input/Output safety guardrails.

**Why LangGraph?**
Unlike traditional linear LLM pipelines, onboarding requires multi-turn memory, conditional routing (e.g., checking caches first, deciding if user input is out of scope, escalating to human support), and state preservation. LangGraph allows defining clear, deterministic nodes (actions) and conditional edges (decision branches) wrapped in a structured state dict.

---

### Q2: Can you describe the request lifecycle step-by-step from user query to response?
**Answer:**
Every message goes through the compiled LangGraph workflow:
1. **Rate Limiting**: Checks if the user has exceeded their request quota (handled in Redis).
2. **0. Cache Node**: Checks L1 Redis cache (exact matches) and L2 ChromaDB cache (semantic similarity matches). If a hit occurs, it routes directly to the **Respond Node** (under 5ms latency).
3. **Input Guardrails**: Evaluates if the query is safe (no toxicity, no prompt injections) and ensures the cumulative session cost does not exceed the `$0.50` budget.
4. **1. Intent Node**: Classifies the query as `General`, `Followup`, or `OutOfScope` using Gemini. Out of scope queries bypass retrieval and route to the escalation module.
5. **2. RAG Node**: Retrieves the top 10 relevant document chunks matching the user's role from ChromaDB, reranks them to the top 3 using FlashRank, and builds a grounded response. If the database is unreachable, it fires the local keyword fallback scanner.
6. **3. Confidence Node**: Evaluates the semantic confidence of the generated answer. If the score is $< 0.5$, it redirects to the **Escalate Node**.
7. **4. Escalate Node**: Drafts a support ticket with escalation details and generates a structured alert (`ALERT:HR_ESCALATION`).
8. **5. Progress Node**: Scans the interaction against onboarding completion criteria using Regex pattern matching to update the user's progress checklist.
9. **6. Respond Node**: Compresses conversation history, saves high-confidence answers back to the L1/L2 caches, and triggers the output safety guardrail (masking internal secrets and checking grounding before display).

## ⚡ Part 2: Caching, Latency, and Cost Optimization

### Q3: What happens if a user asks the same query repeatedly? Do you call the LLM every time?
**Answer:**
No. We implement a **Two-Tier Hybrid Caching Strategy** in `utils/cache.py` to bypass the LLM and RAG loops for identical or highly similar queries:
1. **L1 (Redis Exact Match - Hashed Key)**: 
   * The user's query is normalized (whitespace stripped, lowercase) and hashed using SHA-256.
   * A lookup is done in Redis. If found, the cached response is served instantly ($<5$ms).
2. **L2 (ChromaDB Semantic Match)**:
   * If L1 misses, we compute the query's vector embedding and search a dedicated cache collection (`onboarding_semantic_cache`) in ChromaDB.
   * If we find a cached query with a **cosine similarity score $\ge 0.92$**, we consider it a hit.
3. **Cache Propagation**:
   * On an L2 cache hit, we save the result back to L1 Redis with a 24-hour TTL so subsequent requests avoid the embedding calculation completely.
   * High-confidence responses generated by the graph ($\ge 0.8$ confidence) are proactively written to both L1 and L2 caches.

---

### Q4: How do you prevent cache pollution or unauthorized data leaks through caching?
**Answer:**
**Problem**: If User A (a new hire) asks a question and triggers an LLM response containing confidential manager data, and User B (another new hire) asks a similar question, User B might receive the cached manager data.
**Resolution**: We implemented **Role-Partitioned Caching**. 
* We suffix all cache keys and metadata in L1 (Redis) and L2 (ChromaDB) with the user's active role (`user_role`).
* E.g., Redis key format: `cache:{user_role}:{query_hash}`.
* This ensures a Joinee's cache hit only pulls answers generated for the `joinee` role, and never accesses answers cached for an `HR` or `Admin` user.

---

### Q5: How did you optimize API embedding costs and latency during cache misses?
**Answer:**
**Problem**: In early designs, the query embedding was computed separately during the L2 cache lookup, and then *computed again* during RAG document retrieval, causing duplicate API requests, high latency (~400ms delay), and high token billing.
**Resolution**: We implemented a **Single-Embedding Execution** pattern:
* We added a `query_vector` field inside the shared LangGraph `AgentState`.
* During L2 Cache checking, we compute the embedding once and store it in `state["query_vector"]`.
* If L2 misses, the RAG Node retrieves this pre-computed vector from the state, bypassing a second API call. This reduced embedding cost by **50%** and cut execution latency by ~200–500ms on cache misses.

## 📚 Part 3: Robust Retrieval-Augmented Generation (RAG)

### Q6: What if ChromaDB or the Gemini embedding service fails? How does the application survive?
**Answer:**
**Problem**: If the vector store is unreachable or the embedding API fails, a standard RAG system crashes or returns blank screens.
**Resolution**: We built a **Resilient Local Fallback Scanner** (`fallback_local_retrieval` inside `rag/retriever.py`):
* The retriever functions are wrapped in `try-except` blocks. If initialization or search fails, a fallback is triggered.
* The system reads a local flat FAQ text file (`data/onboarding_faq.txt`).
* It tokenizes the query, filters out stopwords, and performs a keyword match scan across the document paragraphs.
* Chunks are scored based on matching word overlaps, sorted, and the top 3 matches are returned as context.
* It sets a `degraded` flag to `True` in the state, which displays an warning banner in the Streamlit UI to notify the user they are viewing cached offline backup data.

---

### Q7: How do you ensure users do not retrieve onboarding documents they are not authorized to see?
**Answer:**
We implement **Metadata-Level Role Gating**:
1. During document ingestion (`ingest.py`), chunks are tagged with a `required_role` attribute (e.g., `joinee`, `HR`, `admin`).
2. When querying ChromaDB in `retriever.py`, we construct a dynamic filter query using Chroma's `where` dictionary.
3. If the user is a `joinee`, the filter restricts results to `{"required_role": {"$in": ["joinee"]}}`.
4. If the user is `HR`, the filter expands to `{"required_role": {"$in": ["joinee", "HR"]}}`.
5. Admins bypass the filter. Unauthorized documents are completely hidden at the database query layer, ensuring zero leakages.

---

### Q8: What is FlashRank and why is it used instead of a standard vector search?
**Answer:**
Standard vector database queries calculate cosine distance, which matches semantic similarity but sometimes misses fine-grained relevancy. 
* We retrieve the top 10 chunks from ChromaDB.
* We pass them to **FlashRank**, an ultra-lightweight, CPU-optimized cross-encoder reranking model.
* FlashRank scores the relevance of each document chunk relative to the user's specific query and filters it down to the top 3 highest-quality chunks.
* This dramatically improves response accuracy and grounding while keeping CPU consumption low (avoiding heavy GPU-bound cross-encoders).

## 🛡️ Part 4: Safety, Guardrails, and Concurrency Hardening

### Q9: How do your safety guardrails work, and how did you prevent prompt injection bypasses?
**Answer:**
Safety checks are decoupled into Input and Output validators in `guardrails/guard.py`:
* **PII Leakage**: Scans input with Regex for phone numbers, credit cards, emails, and credentials, redacting them before execution.
* **Toxicity**: Uses a word blacklist to reject queries.
* **Budget Limits**: Tracks session cost and blocks calls if total cost $\ge \$0.50$.
* **Prompt Injection Defense**: Normalizes input strings using Unicode NFKC normalization (`unicodedata.normalize("NFKC", text)`) and strips diacritics/accents. This prevents bypass attempts using homoglyph characters or stylized fonts. It scans for override keyword sequences (e.g., "ignore previous instructions") and rejects the query if found.
* **API Key / Secret Leaks**: Output guardrails scan the generated text using regex. If any internal API key or password is leaked, it blocks the response and alerts administrators.

---

### Q10: How did you handle SQLite concurrency issues and thread safety in Streamlit?
**Answer:**
**Problem**: Streamlit runs on a multi-threaded server. SQLite connections created globally are shared across threads, which leads to `sqlite3.ProgrammingError: SQLite objects created in a thread can only be used in that same thread` or database locks during simultaneous conversations.
**Resolution**: 
1. **ChromaDB Client**: We created a shared thread-safe singleton manager (`utils/chroma_manager.py`) to manage connection requests.
2. **LangGraph Checkpointer**: Instead of sharing a global SQLite connection object, we configured `SqliteSaver` using a connection with `check_same_thread=False` and separate session connections.
3. **Locking**: Ensured connections are handled atomically to avoid blocking reads/writes.

## 🧪 Part 5: MLOps, Quality Testing, and Compliance

### Q11: How do you verify the quality and accuracy of the RAG responses?
**Answer:**
We implement an **Automated Evaluation Pipeline** in `tests/test_evals.py` using **DeepEval**:
* We run automated unit tests using PyTest.
* DeepEval evaluates retrieved context and generated responses using custom metric assertions:
  * **Faithfulness**: Checks if the LLM response is fully grounded in the retrieved chunks without hallucinating.
  * **Answer Relevancy**: Evaluates if the response directly addresses the user's query.
* These metrics utilize a Gemini model as an evaluator, comparing output against a structured benchmarking dataset (`tests/test_dataset.json`).

---

### Q12: How does the feedback loop work in production?
**Answer:**
* When a user clicks 👍 or 👎 on a response in the Streamlit UI, the event triggers a callback to `utils/feedback.py`.
* The rating, query, and generated response are logged inside `feedback_history.db`.
* **Flywheel Effect**: Down-rated queries (thumbs-down) are automatically formatted and appended as new regression test cases in `tests/test_dataset.json`. During the next deployment cycle, the test suite verifies that the updated system does not repeat those errors.
* We also integrate with **LangSmith** to annotate tracing runs with feedback scores, allowing developers to debug logs in real time.

---

### Q13: How is GDPR compliance (Right to Be Forgotten) handled in this project?
**Answer:**
We built a dedicated administrative utility `utils/purge_user.py`:
* Given a user's session ID, the script runs database queries to wipe all conversation state checkpoints, session histories, and feedback ratings stored across SQLite databases (`onboarding_history.db` and `feedback_history.db`).
* A second utility `utils/prune_db.py` is configured as a cron job to automatically delete checkpoints and logs older than 30 days, adhering to standard data retention policies.

## ⚙️ Part 6: System Design, State Management, and Cost Control

### Q14: How does the application prevent prompt context window bloating during long, multi-turn conversations?
**Answer:**
**Problem**: In multi-turn RAG conversations, appending every user query and model answer causes the prompt size to grow continuously. This leads to high API costs, increased response latency, and "lost-in-the-middle" issues where the LLM forgets context.
**Resolution**: We implement **LLM-Powered Context Summarization** in the `respond_node` in `graph/nodes.py`:
* If the conversation history exceeds **10 messages**, the oldest turns (excluding the last 4 turns) are processed by an LLM summarizer.
* The LLM condenses the older history into a compact summary (under 3 sentences) focusing on user preferences, roles, and setup steps.
* The system keeps the last 4 messages fully intact (to maintain immediate context flow) and replaces the old turns with a single system message:
  `{"role": "system", "content": "Summary of earlier conversation: [Concise summary]"}`.
* This maintains the core context while keeping token usage bounded and cost-efficient.

---

### Q15: How does the Onboarding Checklist tracking work under the hood? Does it use LLM agents to determine completion?
**Answer:**
**Problem**: Using an LLM to scan every message and check off tasks (like IT Setup or Payroll completion) is expensive, slow, and non-deterministic.
**Resolution**: We implement a **Deterministic Regex-based Progress Scanner** in `utils/progress.py`:
* We defined a mapping of topics to exact keyword regex patterns (`TOPIC_KEYWORDS`). E.g., `Payroll` checks for `salary`, `tax`, `payslip`, `deductions`, `bank account`.
* When a user submits a query, the `progress_node` executes `update_topics_covered` using standard Python `re.search()`.
* If a pattern matches, the task state is marked `True`.
* By using **pure Python regex rather than LLM classification**, we achieve **0ms latency, zero token costs, and 100% deterministic accuracy** for checklist tracking.

---

### Q16: How do you track and enforce the session budget cost safeguards?
**Answer:**
**Problem**: If a rogue loop or API abuse script interacts with the chatbot, it can rack up hundreds of dollars in API billing.
**Resolution**: We track token metadata in real time:
* In `graph/nodes.py`, the `parse_token_usage` function reads the `usage_metadata` object returned by the Gemini API response.
* It calculates the approximate cost of each API call using standard pricing parameters:
  $$\text{Cost} = \left(\frac{\text{Input Tokens} \times \text{Input Cost}}{1,000,000}\right) + \left(\frac{\text{Output Tokens} \times \text{Output Cost}}{1,000,000}\right)$$
* The cost is accumulated in the shared `AgentState` under `token_usage["total_cost_usd"]`.
* Before any LLM call, the **Input Guardrail** (`guardrails/guard.py`) inspects this cost value. If it exceeds the maximum budget threshold (default: `$0.50`), the query is blocked, the LLM is bypassed, and the user is redirected to the HR Escalation node.

---

### Q17: What happens to the rate limiter or caching layer if Redis goes down?
**Answer:**
**Problem**: A hard dependency on Redis for caching and rate limiting can lead to a single point of failure (SPOF) if the Redis container crashes.
**Resolution**: We implement a **Fail-Open Design Principle** in both `utils/rate_limiter.py` and `utils/cache.py`:
* The Redis client initialization is wrapped in a `try-except` block. If `Redis.ping()` fails, the client is marked as unreachable (`False`).
* In `is_rate_limited()`, if Redis is unreachable, the exception is caught, a warning is logged, and the function returns `False` (fails open). This prevents users from being locked out of the assistant.
* Similarly, in the caching layer, if Redis is down, the L1 exact-match cache is bypassed and the system proceeds directly to check ChromaDB or execute the RAG graph, guaranteeing maximum service uptime.
